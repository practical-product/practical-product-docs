# 프론트엔드 아키텍처

Next.js **App Router** 기반이며, 화면 구성과 로직을 명확히 분리하기 위해
Feature 단위 폴더 구조를 채택했습니다.

## 라우트 구조
app/
├─ (auth)/ # 로그인, 회원가입 등 인증 관련 라우트 그룹
└─ (shop)/
├─ cart/
├─ categories/[id]/
├─ checkout/
├─ mypage/
├─ orders/
├─ payments/
├─ products/
├─ search/
└─ admin/

라우트 그룹 `(auth)`, `(shop)`으로 레이아웃과 접근 성격이 다른 페이지를
URL 구조 변경 없이 분리했습니다.

## 폴더 구조
src/
├─ app/ # 라우팅, 페이지 (App Router)
├─ features/ # 도메인별 화면 로직 (auth, cart, order, payment, product, category, admin)
├─ shared/ # 공통 API 클라이언트, UI, hooks, lib, types, config
└─ widgets/ # 여러 feature를 조합한 상위 UI 블록

- `features/`는 도메인별로 나뉘어 있어 특정 기능(예: 장바구니) 수정 시
  해당 폴더만 보면 됩니다.
- `shared/api/client.ts`는 공통 API 클라이언트, `shared/components`는
  로딩/에러 등 공통 상태 뷰(`StateViews.tsx`)를 포함합니다.

## 상태 관리

- **Zustand**로 장바구니 등 클라이언트 상태 관리 (`features/cart/store.ts`)
- 서버 상태는 API 클라이언트를 통해 관리

## 테스트

- **Playwright**로 E2E 테스트 (`e2e/`, `playwright.config.ts`)
- 클라이언트/스토어 단위 테스트 (`store.test.ts`, `client.test.ts`)

## 패키지 관리

- `pnpm workspace` 기반 모노레포 설정 (`pnpm-workspace.yaml`)
