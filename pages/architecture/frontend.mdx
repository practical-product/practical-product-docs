# 프론트엔드 아키텍처

**MUSTIT** — 무신사 앱처럼 PC든 모바일이든 항상 가운데 모바일 화면으로 보이는
커머스 웹앱입니다. [백엔드](/architecture/backend)의 REST API를 소비하는 클라이언트로,
Next.js **App Router** 기반입니다.

## 모바일 퍼스트 UI

PC 브라우저로 열어도 화면 중앙에 480px 폭의 모바일 프레임이 고정되고,
하단 탭바로 주요 화면을 이동합니다. 무신사·지그재그 같은 모바일 커머스 앱의
UX를 웹에서도 동일하게 재현하는 것이 목표입니다.

## 화면 구성

| 경로 | 설명 |
|---|---|
| `/` | 홈 — 히어로, 카테고리, 신상품 무한스크롤 |
| `/products`, `/products/[id]` | 상품 목록 · 상세, 장바구니 담기 · 바로구매 |
| `/categories`, `/categories/[id]` | 카테고리 트리 · 카테고리별 상품 |
| `/search` | 상품명 검색 |
| `/cart` → `/checkout` | 장바구니 → 주문서 |
| `/payments/success`, `/payments/fail` | 토스페이먼츠 결제 콜백 |
| `/orders`, `/orders/[id]` | 주문내역 · 상세, 구매확정 |
| `/mypage`, `/mypage/edit` | 내 정보 조회 · 수정 · 로그아웃 |
| `/login`, `/signup` | 인증 |
| `/admin/products`, `/admin/categories` | 관리자 CRUD |

## 레이어 구조 (Feature-Sliced)

```text
app → widgets → features → shared
```

상위 레이어만 하위 레이어를 참조할 수 있고, **feature끼리는 서로 참조하지 않습니다.**
예를 들어 `cart` feature가 `order` feature 내부를 직접 import하는 것은 금지되어 있고,
이는 ESLint 규칙으로 강제됩니다. 기능이 늘어나도 서로 얽히지 않도록 하기 위한
설계입니다.

## 서버 상태와 클라이언트 상태를 분리

- **서버 상태** (상품 목록, 무한스크롤 등)는 **TanStack Query**로 관리 — 캐싱, 재요청,
  로딩/에러 상태를 일관되게 처리
- **클라이언트 상태** (인증 토큰, 장바구니, 주문 이력)는 **Zustand**로 관리하며
  `localStorage`에 영속화되어 새로고침해도 유지됨

## 백엔드 제약에 대한 대응

REST API가 아직 커머스 프론트엔드에 필요한 모든 기능을 제공하지 않는 부분은,
프론트엔드에서 임시로 보완하고 있습니다.

- **장바구니 API 부재** → 장바구니를 클라이언트 상태로 구현하고, 결제 시점에
  주문 생성 API로 한 번에 전달
- **주문 목록 조회 API 부재** → 결제가 완료된 주문 ID를 클라이언트에 기록해
  주문내역 화면을 구성
- **상품 이미지 필드 부재** → 상품별로 고정된 그라디언트 플레이스홀더를 생성해 표시

이런 임시 대응은 백엔드 API가 보완되는 대로 점진적으로 걷어낼 예정입니다.

## 결제 연동

체크아웃 화면에서 **토스페이먼츠 SDK**로 결제를 요청하고, 결제 성공/실패 콜백
경로(`/payments/success`, `/payments/fail`)에서 백엔드의 결제 승인 API를 호출해
최종적으로 주문을 확정합니다.

## 폼과 검증

회원가입, 로그인, 배송지 입력 등의 폼은 `react-hook-form` + `zod` 조합으로
작성해, 타입 안전한 검증 로직을 재사용합니다.

## 품질 관리

- PR·push마다 GitHub Actions로 린트 · 타입체크 · 유닛 테스트 · 빌드 · E2E 스모크
  테스트를 자동 실행
- 유닛 테스트는 API 클라이언트 · 장바구니 스토어 · 포맷터 등 핵심 로직 위주
- E2E 테스트(Playwright)로 앱 셸이 정상적으로 뜨는지 스모크 테스트
- 환경변수는 스키마로 검증해, 잘못된 값이면 애플리케이션이 아예 기동되지 않도록 방지
